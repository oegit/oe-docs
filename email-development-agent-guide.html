Email Development Agent:
turn a Figma design into a production-ready HTML email

A portable, self-contained agent that turns Claude Code into an expert HTML email developer. Copy the folder, point it at a project, and it builds emails that render correctly everywhere — including the clients that break everything.

Outlook Windows Gmail Apple Mail Outlook.com Yahoo Mail Dark mode ✓

What it is (30 seconds)

Think of the agent as a senior email developer packed into a folder. It's a set of markdown files: an identity file that tells Claude who it is, plus battle-tested rule files for everything that makes HTML email painful — Outlook quirks, dark mode, responsive behavior. Every lesson from every past project is already written into those rules, so each new project starts with all the scar tissue included.

What's inside the folder

Six knowledge files plus a projects/ folder. Claude reads them automatically at the start of every project — you never open them yourself.

CLAUDE.mdThe agent's identity, tech stack, and core rules. Claude reads it on startup.
OUTLOOK_RULES.mdEverything Outlook breaks and how to fix it — VML buttons, conditional comments, font fallbacks.
DARK_MODE_RULES.mdPer-client strategies to force light mode, with a dated experiment log of what works and what regresses.
RESPONSIVE_RULES.mdMobile behavior: breakpoints, fluid images, stacking rules.
FIGMA_TO_EMAIL_WORKFLOW.mdThe 10-step process from design file to production HTML.
PRE_SEND_QA.mdThe consolidated checklist that gates every real send.

The two project types

Every project declares its type in its own brief (CLAUDE.md). The agent handles both.

Standard

Figma design → 1 production HTML
  • The most common case: one design, one email.
  • No variables, no scripts, no automation.
  • Exact values from the design go straight into the template.

With Variables

Figma design + data source → N HTMLs
  • One template, generated many times with different data.
  • Placeholders like {{CLIENT_NAME}} get filled from a Google Sheet or other source.
  • A generation script produces one email per record or per language.

Use it in any Claude account

The agent is client-agnostic by design: brand colors, fonts, and voice enter through each project's brief — never through the agent itself. That's what makes it portable.

Copy the folder

Grab the entire agent-email-development/ folder and put it on any machine — yours or someone else's.

Open Claude Code pointing to it

Launch Claude Code with that folder selected. Claude reads the agent's CLAUDE.md automatically on startup and becomes the email developer.

Point it at a project

Say: "The project is at [path]. Read its CLAUDE.md and tell me when you're ready." Projects can live inside the agent's projects/ folder or anywhere else.

Work normally

No account memory, no special configuration, no dependency on any workspace. Everything the agent knows travels inside the folder.

How a project runs

The full 10-step workflow, condensed. Claude drives; you review at the checkpoints.

Read the project brief Claude

Project type, client, purpose, data source, documented decisions.

Read the design from Figma Claude

Through Figma's Dev Mode connection, Claude extracts every design token: colors, typography, spacing, dimensions, assets.

Map the design blocks Claude

Every vertical block gets documented before a single line of code — background, padding, content, fonts. Fixed vs variable elements are flagged.

Build and compile the template Claude

The email is written in MJML (the source of truth), compiled to HTML, then hand-finished with the Outlook-specific fixes automatic tools miss.

Generate the batch Claude

With Variables only: the script reads the data source and outputs one HTML per record — with a mock mode for safe testing first.

QA across real clients You + Litmus

The HTML goes to Litmus and gets checked on the priority clients: Outlook Windows first (the most fragile), then Gmail, Apple Mail, Outlook.com, Yahoo — light and dark mode, desktop and mobile.

Fix and document Claude

Fixes get applied — and any new discovery is written back into the rule files. The agent gets permanently smarter with every project.

Pre-send QA gate You

Before any real send, the consolidated checklist runs: structure, fonts, images, CTAs, sign-off. Nothing ships without passing it.

Rules the agent never breaks

A taste of what's baked into the rule files — the reason the output survives Outlook.

01Tables, not divs. All layout uses tables — the only structure every email client respects.
02Arial before Helvetica in every font fallback — Helvetica-first stacks turn into Times New Roman in some Outlook versions.
03Force light mode, always. Emails never adapt to dark mode — each client gets its own documented strategy to keep colors intact.
04Explicit backgrounds everywhere. Every content cell declares its own background color — no exceptions, no surprises.
05Rounded buttons need VML. Outlook ignores CSS border-radius; the agent hand-writes the vector markup that makes buttons round there too.

Quick questions

Does it work for any brand or client?

Yes — that's the design. The agent carries only generic email-development knowledge. Brand rules (colors, fonts, voice, claims) live in each project's own brief, so the same agent serves any client without modification.

What do I need installed?

Claude Code, Node.js, and MJML installed inside the project folder. For design extraction: Figma Desktop with its Dev Mode server enabled. For QA: a Litmus account.

Why MJML instead of writing HTML directly?

MJML generates the bulletproof table structure automatically and stays readable. The agent then hand-finishes the compiled HTML with the client-specific fixes MJML can't know about.

How does the agent improve over time?

Every new bug found in QA gets documented in the rule files with the problem, the client, the fix, and why it works. The next project inherits all of it automatically.

Can I run it without a Figma design?

Yes — Figma is the standard input, but the agent can build from any sufficiently detailed spec. The design-token extraction step is just skipped or replaced by the spec.
