Landing Page A/B Testing:
the operations guide

The system runs A/B tests on landing pages continuously, learning which content drives sign-ups and which drives sales. It mostly runs itself. You step in at five moments — that's the whole job.

No coding needed No Claude experience Dashboard + VWO ~20 min per test

Who does what

Four of your five jobs are a single click in the dashboard. The fifth — publishing the test in VWO — happens outside it and has its own step-by-step runbook.

The agent, on its own

no action needed from you
  • Proposes test ideas (hypotheses)
  • Writes the build brief and builds the variants
  • Runs the quality, brand and SEO checks
  • Watches results and picks the likely winner
  • After you confirm: full SEO pass, promotes the winner, drafts social content

You

five moments, in order
  • Approve which ideas run
  • Approve the finished variant before it goes live
  • Publish the test in VWO, then mark it live
  • Approve a request to run a test longer
  • Confirm the winner

Why you publish the test yourself

This surprises people, so it's worth one paragraph.

VWO only sells API access with its enterprise tier, so nothing can start a campaign on your behalf. This is normal and the system is built for it — you follow a click-by-click runbook and the test goes live in about 20 minutes.

You do not need to fix this, and please don't click "Upgrade" in VWO. Whether OE buys that tier is a commercial decision, not something this system is waiting on.

Getting in

Everything except job 3 happens on one screen.

Open the dashboard

Use the link your admin gives you (for example http://<host>:4317).

Enter the shared password

Your admin has it. You'll land on Overview.

The four screens

OverviewEvery test: status, market, goal, and how many days it's been running.
ApprovalsYour to-do list — everything waiting on a human. The number beside it is how many are pending.
Test DetailOne test: its idea, the variants, results so far, and the winner to confirm when it's ready.
SettingsAdmins only — which steps need a human vs. run automatically. Leave as-is unless told otherwise.

Your five jobs

In the order they happen. Jobs 1, 2, 4 and 5 are one click. Job 3 is the one that isn't.

Approve a hypothesis one click

A new test idea.

  1. Go to Approvals.
  2. Find the card tagged HYPOTHESIS and read the idea.
  3. Approve to let it become a test, or Reject to drop it.

Approve a creative one click

A finished variant, before it goes live.

  1. In Approvals, find the card tagged CREATIVE.
  2. It means a variant was built and passed the quality + SEO checks.
  3. Approve to send it to publication, or Reject to send it back.

Publish the test in VWO ~20 min, in VWO

The test sits marked awaiting-manual-publish until you do this. The agent will never claim a test is live on its own — that wait is deliberate.

  1. Open the Variant Build Brief for the test (linked from Test Detail). Everything you type comes from it.
  2. Follow the VWO runbook step by step — runbooks/VWO_MANUAL_RUNBOOK.md in the agent folder. It's written for someone who has never opened VWO.
  3. When the campaign reads Running in VWO, return to the dashboard, mark the test live, and record the VWO campaign ID.

If a field in the brief is blank, stop and send it back. Don't invent a value.

Approve a duration change one click

The agent is asking to extend a test, usually to gather a full week of data.

  1. In Approvals, find the card tagged DURATION CHANGE.
  2. Approve to extend, or Reject to keep the original end date.

Confirm a winner one click

The final say is always yours.

  1. In Approvals, find the card tagged WINNER — or open the test from Overview.
  2. On Test Detail you'll see both variants, their results, and the one the agent picked (marked ★).
  3. If it looks right, click Confirm winner. If not, leave it — the test keeps running.

Nothing bad happens by waiting. When unsure, ask your admin.

The one mistake that matters

Most errors announce themselves. This one doesn't — it produces a confident, wrong answer.

VWO needs its tracking snippet (SmartCode) on every page in the test — including a variant hosted at its own URL. A separately built page does not get it automatically.

If it's missing, VWO still counts visitors but counts zero conversions. So the test looks perfectly healthy, data accumulates, and the variant loses — for no reason other than a missing script tag. The loss then reads like a real finding about your idea.

The runbook shows you how to check it with VWO's own snippet checker, one URL at a time. Don't skip that check, and run it on the variant's URL too, not just the control's.

What VWO's "95%" actually means

Worth thirty seconds, because it's easy to misread — and easy to repeat incorrectly to someone else.

VWO calls a variation a Winner when its "Probability to be best" is at least 95%. That is not the same as "95% confidence" or "statistically significant", even though it looks like it.

It's the chance that this variation really is the best one — a different measurement that happens to be written as a percentage. When you report a result, use VWO's words ("96% probability to be best"), not "96% confident".

And VWO calling a winner doesn't mean the test is done. We additionally require full weekly cycles — a test that hits 95% on a Tuesday has only seen Tuesday-shaped traffic. VWO also won't call anything before 25 conversions per variation, 1,500 visitors, and a week of running.

Never promote a page just because VWO put a trophy on it. That's what your confirm step is for.

After you confirm a winner

Automatic, with no further action from you.

The winning page gets a full SEO pass and is promoted

It becomes the live page.

The winning message becomes draft social content

Video, posts, carousels, emails.

Those drafts wait for a human

Nothing auto-publishes. Someone reviews before anything ships.

The agent proposes the next test idea

And the cycle repeats — a closed test immediately seeds the next.

When to escalate

→A variant is getting visitors but zero conversions. Almost always the missing SmartCode. Tell your admin — don't wait for it to "settle".
→A test looks broken, results seem impossible, or a variant renders wrong. Reject and tell your admin.
→A menu name in the VWO runbook doesn't match your screen. Trust your screen, then tell your admin so the runbook gets fixed. VWO is mid-way through changing its interface.
→You're unsure whether to confirm a winner. Leave it and ask. Nothing bad happens by waiting.
→Password reset, or the dashboard won't load. Your admin.

Questions

Do I need to understand the statistics?

No. The agent evaluates the rule and preselects a winner; you confirm. The only thing worth knowing is that VWO's "95%" isn't "95% confidence" — so when you pass a result along, quote VWO's wording rather than translating it.

Why is publishing manual when everything else is automatic?

VWO sells API access only with its enterprise tier, so no software can start a campaign for us. The system was designed around this: it's manual at exactly one step, never blocked. If OE ever gets API access, that step disappears and nothing else changes.

How many tests run at once?

Two by default — usually one lead-gen page and one sales page. Traffic gets split between the tests that are running, so more tests means slower results. Extra ideas queue instead of starting.

Can I change the copy while I'm building the variation in VWO?

No. The copy in the brief is final and already localized. Anything you change beyond what the brief lists becomes part of what's being tested, and the result gets attributed to the wrong cause. If the copy looks wrong, raise it before launch.

Something in the dashboard says "confidence". Is that the same as VWO's number?

Not exactly, and it's being corrected. Treat VWO's own report as the source of truth for what the number measures, and ask your admin if a figure looks off.

Notes for the admin

  • The dashboard is served by one command (node serve.js) and needs no internet to run.
  • The password and every integration are set per deployment in PROJECT_CONFIG; secrets stay outside the code as pointers.
  • Settings → decision-rights: each of the four approvals can be switched to "auto" if you fully trust that step. Start with all set to human. Job 3 (publishing) is not a toggle — it's manual because VWO's API is enterprise-only.
  • testingTool.integration is manual, pointing at runbooks/VWO_MANUAL_RUNBOOK.md. If VWO API access ever lands, an automated adapter slots in beside the runbook and job 3 disappears — nothing else changes.
  • The runbook marks the labels it could not verify from VWO's docs with ⚠️. If an operator reports a mismatch, fix it there.
