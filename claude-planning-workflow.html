The pipeline: from idea to production

Five stops. Each one happens in a different place — the chip tells you where.

STOP 01

Define the idea

Your head + notes

Write down in 5 minutes: the objective, what's in scope, what's NOT in scope, and the constraints. Without this, everything else gets built on sand.

STOP 02

Plan

Chat (Claude app)

Say "let's plan X" and attach every piece of evidence you have — the planning-kickoff skill loads the right template automatically. No GitHub access needed here: it's a regular chat. The output is a phased plan you download: PLAN_X.md.

Why in chat: planning is cheap; fixing a poorly planned execution is expensive. A planning mistake multiplies into every task built on top of it.
STOP 03

Verify

Claude Code

This is the first stop that touches the real repo. Use the Claude Code template with a high-effort model and Plan Mode ON. Claude reads the real files and confirms or rejects every assumption in the plan (✅ / ❌ / ⚠️), then the plan lands in audit/.

Why verify: chat can't see your files. The discoveries that save projects (hidden repos, sensitive data, stale configs) only show up when the real repo gets read.
STOP 04

Execute

Claude Code

Task by task, in dependency order. One commit per task, with a verification step before moving to the next. You give the "go" at every risky step.

STOP 05

QA and dry run

Claude Code

Test everything in a safe environment before touching anything that's already in production. If the dry run fails, it gets fixed here — never in production.

Allowed shortcut: if the task is tiny (a typo, a banner, a small tweak), skip stops 02 and 03 and go straight to execution. Formal planning for a 10-minute task is bureaucracy, not rigor.

Where everything gets documented

Every document has a fixed home. If you don't know where to look for something, this table is the answer.

DocumentWhere it livesWhat it's for
Planning templates Installed as the planning-kickoff Claude skill — source mirrored at oe-docs/downloads/planning-kickoff/ The reusable prompts that kick off any planning session. As an installed skill, they load automatically — nobody opens or pastes them by hand.
Plans (PLAN_X.md) audit/ inside the repo The phased plan for each project. Versioned in Git — it's the decision history.
Audits (AUDIT_X.md) audit/ inside the repo What was found when reading the real state before planning or executing.
How-to guides Online, this site (oe-docs) How to use the skill and the workflow. This is the version that gets shared with the team.
Foundational context Foundation files at the workspace root CLAUDE.md, BRAND_GUIDELINES.md, COMPANY_CONTEXT.md, PROJECT_STRUCTURE.md, PROMPTING_STANDARDS.md — rules that apply to every project and agent.
Project-specific context Each project/agent's own folder Rules and identity that only apply there — e.g. the email agent's OUTLOOK_RULES.md, or a project's own brief. Never duplicated into the foundation files.
Only Stop 03 onward needs repo access. Stop 02 (Plan) is a regular chat — evidence you attach plus the skill is enough. Repo access only matters once someone verifies or executes.

The two templates, side by side

They are not two versions of the same document — they are two different moments in the pipeline. Both now ship inside the planning-kickoff skill, so you don't paste them manually anymore.

Chat template

PLANNING_TEMPLATE_CHAT.md
When
Stop 02 — to turn basic specs into a solid plan.
Where
Chat in the Claude app (or inside a Project). Loads automatically once the skill is installed.
Model
The most capable model available, with extended thinking on.
Key rule
The model can't see your files: everything unconfirmed gets tagged [UNVERIFIED].
Produces
PLAN_X.md + a handoff prompt ready for Claude Code.

Claude Code template

PLANNING_TEMPLATE_CLAUDE_CODE.md
When
Stop 03 — to ground the plan in the reality of the repo.
Where
Claude Code in the Desktop app, opened from the project subfolder.
Model
High effort, Plan Mode ON (it can't touch files until you approve).
Key rule
It verifies every [UNVERIFIED] claim against real files before writing the final plan.
Produces
A verified plan in audit/ + task-by-task execution, one commit per task.

Trigger: just say "let's plan X", "help me plan X", or "new project: X" in any chat with the skill installed — see the planning-kickoff skill guide for the full install + trigger flow.

Quick rules

Is planning the most important part?
Yes, together with verification. Solid plan + verification against real files = a project that ships. Solid plan without verification = a pretty plan that breaks at task 2.
Should everything start inside a Project in the Claude app?
No. Use a Project when you'll have 3+ chats on the same topic (the context stays loaded). Use a regular chat for one-off planning: the skill plus your attached evidence is enough.
Which model for which job?
The axis is risk + ambiguity, not "planning vs. execution."
ModelEffortUse it for
Claude Fable 5—Deciding, planning, audits — deep analysis with lots of evidence
Claude Opus 4.8HighRisky execution: production, new features, rewrites
Claude Sonnet 5MaxRoutine, well-specced code generation
Claude Sonnet 5Low–MediumGit ops and other automation
Claude Haiku 4.5LowMechanical, repetitive tasks
Rough production mix: ~70% Haiku, 20% Sonnet, 8% Opus, 2% Fable. Model names and lineups change — if this looks stale, just ask Claude to refresh it.
When should you NOT plan?
Tasks under ~30 minutes with an obvious scope. Going straight to execution isn't skipping the process — it's applying the right-sized process for the task.
