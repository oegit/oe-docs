The pipeline: from idea to production

Five stops. Each one happens in a different place — the chip tells you where.

STOP 01

Define the idea

Your head + notes

Write down in 5 minutes: the objective, what's in scope, what's NOT in scope, and the constraints. Without this, everything else gets built on sand. Small projects define this inline in the kickoff message (Scenario A). Large projects grow this stop into a full spec document using the process below, and then kick off with Scenario B.

The Spec Doc process — large projects only
  1. Initial dump. Describe the idea in natural language, plus the deliverables you have in mind. No structure required yet — just get it all out.
  2. Expansion. The model proposes everything the dump is missing: undecided points, edge cases, dependencies, hidden assumptions. You decide what makes it into the spec — nothing enters without your call.
  3. Decision rounds. Each round closes a batch of decisions and rewrites the FULL document as a new version (v0.1 → v0.2 → … → v0.N) — never patches. Misunderstandings get corrected on the spot, and technical decisions can be explicitly delegated ("I'll go with your recommendation").
  4. Close. The spec is FINAL only when open questions = 0. Anything still open means another round, not a final spec.
  5. Ship. Upload the final spec to the claude.ai Project's Context panel and launch the Scenario B kickoff.
Rules of the game:
  • Most capable model available, maximum effort — spec mistakes multiply into every downstream task.
  • One versioned doc. The chat is disposable; the doc is the deliverable.
  • Only closed decisions plus explicit open questions — zero loose TBDs inside the text.
  • Spec docs in English, like every Lab document.
  • Prefer running the process inside the Project itself, so the Project's Memory absorbs the decisions as they close.
STOP 02

Plan

Chat (Claude app)

Say "let's plan X" and attach every piece of evidence you have — the planning-kickoff skill loads the right template automatically. No GitHub access needed here: it's a regular chat. The output is a phased plan you download: PLAN_X.md.

Why in chat: planning is cheap; fixing a poorly planned execution is expensive. A planning mistake multiplies into every task built on top of it.

Two kickoff scenarios. Pick by project size: a small project defines everything inline in the message (A); a large project builds a spec document first, and the kickoff message only points to it (B).

Scenario A — small project, no spec doc
Let's plan [PROJECT NAME].

Objective: [...]
In scope: [...]
Out of scope: [...]
Constraints: [...]

The output of this workstream will live in this claude.ai Project (Context + Instructions).
  • Objective — the outcome this must achieve, and why now. 1–3 sentences.
  • In scope — what the project WILL deliver. Concrete pieces, e.g. "landing page + 3 email variants + UTM tracking".
  • Out of scope — what it deliberately will NOT touch, even if related. Being explicit here is what prevents scope creep, e.g. "no CRM changes, no paid media".
  • Constraints — the non-negotiables the plan must respect: deadlines, budget, brand rules, frozen repos, required approvals.

Replace every [...] with your own content before sending — never send the brackets as-is, and delete any line that doesn't apply. The last line only belongs when you're planning inside a claude.ai Project. Attach all your evidence in the same message.

Scenario B — large project with a spec doc
Let's plan [PROJECT NAME].

- Objective, in scope, out of scope and constraints: everything is
  defined and closed in [SPEC_FILE].md ([version]), uploaded to this
  Project's Context. Use it as the complete Define — do not re-ask
  what's already decided.
- The output of this workstream will live in this claude.ai Project
  (Context + Instructions).

Use the planning-kickoff skill. Output: PLAN_[PROJECT NAME].md in English.
  • Spec doc first — build the complete Define (objective, scope, constraints, every closed decision) as one document BEFORE this message, using the Spec Doc process in Stop 01. The doc is the single source of truth; the message only points to it.
  • Upload to Context — on the Project page, use the + button on the Context panel. Every chat in the Project inherits it automatically.
  • One version only — only the current spec lives in Context. When the spec changes, delete the old file and upload the new one, so stale rules can't leak into new chats.
  • Never retype the fields — duplicating spec content inside the message desynchronizes the source of truth.

Use Scenario B whenever a Define/spec document exists. Scenario A is the fallback for projects small enough to define inside the message itself.

STOP 03

Verify

Claude Code

This is the first stop that touches the real repo. Use the Claude Code template with a high-effort model and Plan Mode ON. Claude reads the real files and confirms or rejects every assumption in the plan (✅ / ❌ / ⚠️), then the plan lands in audit/.

Why verify: chat can't see your files. The discoveries that save projects (hidden repos, sensitive data, stale configs) only show up when the real repo gets read.
STOP 04

Execute

Claude Code

Task by task, in dependency order. One commit per task, with a verification step before moving to the next. You give the "go" at every risky step.

STOP 05

QA and dry run

Claude Code

Test everything in a safe environment before touching anything that's already in production. If the dry run fails, it gets fixed here — never in production.

Allowed shortcut: if the task is tiny (a typo, a banner, a small tweak), skip stops 02 and 03 and go straight to execution. Formal planning for a 10-minute task is bureaucracy, not rigor.

Quick rules

"Let's build the spec" vs. "let's plan" — which comes first?
Large projects: spec first, then plan. "Let's build the spec for X" runs the Spec Doc process (Stop 01) until open questions = 0 → the final SPEC_X.md goes into the Project's Context → then, in a new chat, "let's plan X" with the Scenario B kickoff produces PLAN_X.md. Small projects skip the spec entirely: straight to "let's plan X" with Scenario A.

They answer different questions: the spec decides WHAT we're building (closed decisions); the plan decides HOW and in what order (phased tasks, verification, commits). The plan never re-decides what the spec already closed — the bridge between them is the spec in Context, which the kickoff message only points to.

Practical rule: if you can define the whole project in one message → Scenario A. If it needs several decision rounds → spec first (B). A new banner goes straight to plan; a full brand-guidelines rebuild gets a spec.
Is planning the most important part?
Yes, together with verification. Solid plan + verification against real files = a project that ships. Solid plan without verification = a pretty plan that breaks at task 2.
Should everything start inside a Project in the Claude app?
No. Use a Project when you'll have 3+ chats on the same topic (the context stays loaded). Use a regular chat for one-off planning: the skill plus your attached evidence is enough.
What does the full flow look like inside a Project?
If the topic earns a Project, create it empty and plan inside it from chat 1:
Chat 1 → the skill fires → clarifying questions in one batch → you answer → PLAN_X.md is generated in that same chat → upload it to the Project's Context panel with the + button → write the Project Instructions → every new chat in that Project starts with the plan and rules already loaded. For large projects, the spec doc goes into Context BEFORE chat 1 (Scenario B).
Which model for which job?
The axis is risk + ambiguity, not "planning vs. execution."
ModelEffortUse it for
Claude Fable 5HighDeciding, planning, audits — deep analysis with lots of evidence
Claude Opus 4.8HighRisky execution: production, new features, rewrites
Claude Sonnet 5MaxRoutine, well-specced code generation
Claude Sonnet 5Low–MediumGit ops and other automation
Claude Haiku 4.5LowMechanical, repetitive tasks
Rough production mix: ~70% Haiku, 20% Sonnet, 8% Opus, 2% Fable. Model names and lineups change — if this looks stale, just ask Claude to refresh it.
When should you NOT plan?
Tasks under ~30 minutes with an obvious scope. Going straight to execution isn't skipping the process — it's applying the right-sized process for the task.
