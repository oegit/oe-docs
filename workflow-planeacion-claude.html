El pipeline: de idea a producción

Cinco paradas. Cada una ocurre en un lugar distinto — el chip indica dónde.

PARADA 01

Definir la idea

Tu cabeza + notas

Escribe en 5 minutos: objetivo, qué entra en el alcance, qué NO entra, y restricciones. Sin esto, todo lo demás se construye sobre arena.

PARADA 02

Planear

Chat (claude.ai)

Usa el Template Chat con Fable 5 (o Opus para ahorrar créditos). Pega toda la evidencia que tengas. El resultado es un plan por fases: PLAN_X.md.

Por qué en chat: planear es barato, corregir una ejecución mal planeada es caro. Un error de plan se multiplica en cada tarea que se construye encima.
PARADA 03

Verificar

Claude Code

Usa el Template Claude Code con Opus effort High y Plan Mode ON. Opus lee los archivos reales del repo y confirma o tumba cada supuesto del plan (✅ / ❌ / ⚠️).

Por qué verificar: el chat no ve tus archivos. Los hallazgos que salvan proyectos (repos ocultos, datos sensibles, configs viejas) solo aparecen leyendo el repo real.
PARADA 04

Ejecutar

Claude Code

Tarea por tarea, en orden de dependencia. Un commit por tarea, con verificación antes de pasar a la siguiente. Tú das el "go" en cada paso riesgoso.

PARADA 05

QA y dry run

Claude Code

Probar en seco antes de tocar cualquier cosa que ya esté en producción. Si el dry run falla, se corrige aquí — nunca en producción.

Atajo permitido: si la tarea es chiquita (un typo, un banner, un ajuste puntual), sáltate las paradas 02 y 03 y ve directo a ejecutar. Planeación formal para tareas de 10 minutos es burocracia, no rigor.

Dónde queda documentado cada cosa

Cada documento tiene una casa fija. Si no sabes dónde buscar algo, esta tabla es la respuesta.

DocumentoDónde vivePara qué sirve
Templates de planeación OE Claude Lab/templates/ Los prompts reutilizables para arrancar cualquier planeación (Chat y Claude Code).
Planes (PLAN_X.md) audit/ dentro del repo del Lab El plan por fases de cada proyecto. Queda versionado en Git — es el historial de decisiones.
Auditorías (AUDIT_X.md) audit/ dentro del repo del Lab Lo que se encontró al leer el estado real antes de planear o ejecutar.

Contexto permanente Foundation files en el root del Lab CLAUDE.md, BRAND_GUIDELINES.md, COMPANY_CONTEXT.md — las reglas que aplican a todo.

Los dos templates, lado a lado

No son dos versiones del mismo documento: son dos momentos distintos del pipeline.

Template Chat

PLANNING_TEMPLATE_CHAT.md
Cuándo
Parada 02 — para convertir specs básicas en un plan sólido.
Dónde
Chat en claude.ai (o dentro de un Project).
Modelo
Fable 5 con extended thinking. Opción económica: Opus, mismo template.
Clave
El modelo no ve tus archivos: todo lo no confirmado queda marcado [UNVERIFIED].
Produce
PLAN_X.md + un handoff prompt listo para Claude Code.

Template Claude Code

PLANNING_TEMPLATE_CLAUDE_CODE.md
Cuándo
Parada 03 — para aterrizar el plan en la realidad del repo.
Dónde
Claude Code en la Desktop app, abierto desde el subfolder del proyecto.
Modelo
Opus, effort High, Plan Mode ON (no puede tocar archivos hasta que apruebes).
Clave
Verifica cada [UNVERIFIED] contra los archivos reales antes de escribir el plan final.
Produce
Plan verificado en audit/ + ejecución tarea por tarea, un commit por tarea.

Reglas rápidas

¿La planeación es la parte más importante?
Sí, junto con la verificación. Plan sólido + verificación contra archivos reales = proyecto que sale. Plan sólido sin verificar = plan bonito que se rompe en la tarea 2.
¿Todo debe empezar en un Project de claude.ai?
No. Project cuando vas a tener 3+ chats sobre el mismo tema (el contexto queda cargado siempre). Chat normal para planeaciones puntuales: template + evidencia pegada es suficiente.
¿Qué modelo uso para cada cosa?
Fable 5: planeaciones complejas con mucha evidencia. Opus: planeación económica en chat, y verificación/ejecución en Claude Code (effort High). Sonnet: generación de código y automatización rutinaria.
¿Cuándo NO planear?
Tareas de menos de ~30 minutos con alcance obvio. Ir directo a ejecutar no es saltarse el proceso: es aplicar el proceso correcto para el tamaño de la tarea.
